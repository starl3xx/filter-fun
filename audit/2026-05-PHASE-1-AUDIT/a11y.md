## HIGH

### [A11y] Form inputs set `outline: "none"` without focus-visible replacement
**Severity:** High
**Status:** ✅ FIXED (`audit/a11y-high-batch-5`)
**Files:** packages/web/src/components/launch/LaunchForm.tsx:151, components/admin/{BagLockCard.tsx, RecipientForm.tsx, MetadataForm.tsx, AdminTransferForms.tsx}
**Spec ref:** WCAG 2.4.7

**Description:** Multiple form inputs disable the default focus outline without providing a `:focus-visible` replacement (no border-color change, no box-shadow). Keyboard-only users cannot tell which field is focused.

**Recommendation:** Add a `:focus-visible` rule that sets `outline: 2px solid var(--cyan)` or equivalent box-shadow ring.

**Resolution:** Single shared `:focus-visible` rule added to `packages/web/src/app/globals.css` covering `input`, `textarea`, `select`. Uses `outline: 2px solid var(--cyan) !important` (the `!important` is necessary because each form's inline `outline: "none"` outranks pseudo-class rules in the cascade — without it the inline reset would win when `:focus-visible` matches). Inline base resets retained on the form components so mouse focus stays clean. Regression: `packages/web/test/regression/formFocusVisible.test.tsx` reads `globals.css` from disk and asserts the rule exists with the expected outline + offset, plus a render-side guard that the LaunchForm inputs still carry the inline `outline: none` so the two halves stay in sync.

**Effort:** S

### [A11y] SAFE status pill has `icon: null` — violates ARENA_SPEC §12 icon+colour rule
**Severity:** High
**Status:** ✅ FIXED (`audit/a11y-high-batch-5`)
**Files:** packages/web/src/components/arena/StatusBadge.tsx:48
**Spec ref:** ARENA_SPEC §12, §3.3

**Description:** Spec §12: "Status pills carry both color and icon, never color alone." The SAFE case returns `icon: null`, conveying status by colour alone — fails for colour-blind users.

**Evidence:**
```ts
case "SAFE": return {color: C.green, label: "Safe", icon: null};
```

**Recommendation:** `icon: "✓"` (or other safe symbol). Render the icon span unconditionally.

**Resolution:** SAFE returns `{color: C.green, label: "Safe", icon: "✓"}` (U+2713 CHECK MARK — colour-independent, not the emoji ✅ which renders as a coloured photo character on some platforms the same way 🔻 collided with the red CSS colour in H-Arena-3). The `treatmentFor()` return type tightened from `icon: string | null` to `icon: string`, and the rendering site changed from `{icon ? <span>{icon}</span> : null}` to an unconditional `<span aria-hidden>{icon}</span>` so future statuses can't reintroduce the bug. Regression: `packages/web/test/regression/statusBadgeIconContract.test.tsx` pins the four-status icon contract (FINALIST 🏆 / SAFE ✓ / AT_RISK ▼ / FILTERED ▼), asserts the icon span renders unconditionally, and locks the U+2713 codepoint specifically.

**Effort:** XS

### [A11y] Links inputs use `aria-label` instead of visible labels
**Status:** ✅ **FIXED** in `audit/polish-a11y` (Polish 5 — Audit M-A11y-2; bugbot follow-up tightened the placeholders). The 3 link inputs (Website / X·Twitter / Farcaster) are now wrapped in a new `<LinkField>` helper that renders a visible mono-uppercase label above the input AND ties them via `useId()` for the htmlFor / id pair. The legacy `aria-label` attributes are removed (the visible label is the canonical accessible name now). Placeholders are hint-only: "yourdomain.com" / "handle" / "handle". Bugbot caught the earlier draft that used `@handle` for Twitter and Farcaster — `validateLaunchFields` rejects values starting with `@` ("Twitter handle without the @ please."), so the placeholder was actively guiding users toward a validator-rejected format. Pinned by `polishA11yPass.test.tsx` (4 tests covering `getByLabelText` reachability, the legacy `aria-label` removal, the unique useId-generated htmlFor values, and the no-`@`-prefix placeholder constraint).

**Severity:** Medium → Low
**Files:** packages/web/src/components/launch/LaunchForm.tsx:216,224,232
**Spec ref:** WCAG 1.3.1

**Description:** Visible labels benefit users with cognitive/visual disabilities and improve UX for everyone. aria-label is a fallback, not a replacement.

**Recommendation:** Wrap each input in a `<label>` with visible text.

**Effort:** S

## LOW

### [A11y] aria-live region scope unclear on /launch
**Status:** ✅ **FIXED** in `audit/polish-a11y` (Polish 5 — Audit L-A11y-1; bugbot follow-up tightened the role). The page-level `<NoticeCard>` already carried `aria-live="polite"`, but the LaunchForm's `<ErrorNotice>` (used for both field-validation errors and post-pin / post-tx errors) sat outside that region — silent for screen-reader users. Post-fix: ErrorNotice carries `role="status"` + `aria-live="polite"`. Bugbot caught the earlier draft that used `role="alert"` with explicit `aria-live="polite"`: `role="alert"` carries an implicit `aria-live="assertive"`, and overriding with polite produces inconsistent SR behaviour. The spec-correct pairing for polite announcements is `role="status"` (implicit polite). Pinned by `polishA11yPass.test.tsx` (2 tests: the post-fix shape + a negative test asserting the rejected `role="alert"` is no longer on the ErrorNotice surface).

**Severity:** Low
**Files:** packages/web/src/app/launch/page.tsx
**Spec ref:** WCAG 4.1.3

**Recommendation:** Confirm `aria-live="polite"` wraps the error/success regions; if errors render outside it, add an explicit live region.

**Effort:** XS
