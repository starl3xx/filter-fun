## Verdict — TL;DR

The engine's *formula structure* is mostly aligned with the locked spec, but with substantive divergences and a much larger gap underneath: **the indexer's projection from raw rows → `TokenStats` is a no-op stub** (`packages/indexer/src/api/hp.ts:43`). Until that lands, the formulas are functionally inert in production — every token in the cohort scores zero on every component except holderConcentration (which itself returns 0 because `holderBalances` is never populated). Track E exercises the formulas via Python on real corpus data; the TS engine has been validated structurally but never run end-to-end on real Base data.

Phase 2 of this epic CAN still ship the formula lock (named constants, normalization mode, slot-fairness fix, version bump, fixtures, finality field). But "engine matches spec" is only end-to-end true once the indexer projection lands. That projection is **out of scope for the dispatch as written** and should either (a) be added as a sub-epic of 1.22 or (b) be filed as a follow-up that gates mainnet activation independently of the formula lock.

## 1. Velocity (§6.4.1)

**Locked spec (per dispatch):**
- `VELOCITY_LOOKBACK = 96h`
- `VELOCITY_DECAY_TAU = 24h`
- `VELOCITY_PER_WALLET_CAP = 10 WETH`

**Engine implementation:** `packages/scoring/src/score.ts:283-344` (`computeVelocity`).

**What matches:**
- Time decay applied per-buy and per-sell (`score.ts:307`, `score.ts:316`).
- Net inflow per wallet, clamped at zero (`score.ts:337`).
- Churn-window detection: a sell within `churnWindowSec` of a same-wallet buy doubles the sell's decayed weight (`score.ts:325-328`).

**What diverges:**

1. **Magic numbers, not named constants.** `velocityHalfLifeSec`, `walletCapFloorWeth`, `churnWindowSec` live in `ScoringConfig` defaults (`types.ts:281-284`) without spec-named identifiers (`VELOCITY_LOOKBACK`, `VELOCITY_DECAY_TAU`, `VELOCITY_PER_WALLET_CAP`). No `VELOCITY_LOOKBACK` exists in code.

2. **Decay is half-life, not exponential τ.** Engine uses `Math.pow(0.5, age / halfLife)` with `velocityHalfLifeSec = 86400` (24h half-life). The dispatch cites `VELOCITY_DECAY_TAU = 24h`, which conventionally means `exp(-Δt / 24h)`. Half-life of 24h ≈ τ ≈ 34.66h (`24 / ln(2)`). **This is a factor-1.44 difference in the decay rate** — e.g. a 24h-old buy is weighted at 0.5 under the engine's half-life model and at 0.368 under exponential τ=24h. Need spec clarification on which interpretation §6.4.1 intends; the locked-formula PR text needs to nail this.

3. **No 96h hard lookback.** Engine considers every buy/sell in `t.buys` / `t.sells` regardless of age — decay alone bounds the contribution. With τ=24h or half-life=24h the contribution at h=96 is `≈6.25%` — not zero. The dispatch's fixture `04-finals-phase-narrowing.json` expects "lookback narrows to 24h-back at h120" and "pre-cut velocity excluded" — that requires a hard window cutoff, which the engine doesn't implement.

4. **Per-wallet cap is logarithmic flattening, not absolute cap.** Engine: `net / log2(1 + walletTotal/floor)` (`score.ts:340-341`). Spec: `VELOCITY_PER_WALLET_CAP = 10 WETH` → `min(net, 10)`. The fixture `02-single-whale-at-cap.json` expects `log(11) ≈ 2.4` for a 100-WETH whale — that's the `log2(1 + 100/0.001) ≈ log2(100001) ≈ 16.6` shape, **NOT** spec's `min(100, 10) = 10`. The fixture's expected value (`log(11) ≈ 2.4`) appears to follow `ln(net + 1)` semantics — different again. Need to align: spec text + fixture expected value must match the engine implementation exactly.

5. **Churn-boost factor of 2 is the engine's invention.** `score.ts:328` doubles the sell's weight when within churn window. The locked spec sections this audit can read (§6.4.1) don't specify this multiplier. Either a spec amendment or removal — flag.

**Severity:** medium-high. Velocity is 30% of HP; a misaligned formula here drops Track E refit ρ.

## 2. Effective Buyers (§6.4.2)

**Locked spec (per dispatch):**
- `EFFECTIVE_BUYERS_LOOKBACK = 96h`
- `EFFECTIVE_BUYERS_DUST_THRESHOLD = 0.001 WETH`

**Engine implementation:** `score.ts:348-360` (`computeEffectiveBuyers`).

**What matches:**
- Sqrt dampening (`score.ts:352`) per §6.4.2 spec preference. Track E v4 validated `sqrt`.
- Dust threshold filter (`score.ts:356`).

**What diverges:**

1. **Dust floor is 0.005 WETH (5e15 wei), not spec 0.001 WETH.** `types.ts:284` — `buyerDustFloorWeth: 5_000_000_000_000_000n`. Spec says 0.001 WETH (1e15 wei). Easy fix; impacts `03-sybil-swarm-dust.json` fixture expected output.

2. **No 96h lookback.** Engine sums `t.volumeByWallet` (cumulative across the token's whole life), not "buys in the last 96h." Volume is whatever the indexer hands it. To match `EFFECTIVE_BUYERS_LOOKBACK = 96h`, the indexer's `volumeByWallet` must be windowed — currently it's empty (see finding #15) so it doesn't matter today, but the projection needs to be windowed correctly.

3. **No EXCLUDED_TRADERS filter.** Routers / aggregators / MEV-bot list (per §6.4.7) — no such filter exists. Spec example fixture `03-excluded-trader-filter.json` requires it.

**Severity:** medium. Easy mechanical fixes once the constants land.

## 3. Sticky Liquidity (§6.4.3)

**Locked spec (per dispatch):**
- `LP_PENALTY_WINDOW = 24h`
- `LP_PENALTY_TAU = 6h`
- Score = `TVL_TERM − WITHDRAWAL_PENALTY` with amount-weighted age

**Engine implementation:** `score.ts:363-370` (`computeStickyLiquidity`).

**What matches:**
- `avgLiquidityDepthWeth` (TVL term) read from input.
- A withdrawal penalty term is multiplied in.
- Penalty is bounded at 1.0 so score floor is 0.

**What diverges:**

1. **No `exp(-Δt / 6h)` decay over the 24h window.** Engine takes a single aggregate `recentLiquidityRemovedWeth` from input — no time-resolved withdrawal stream. Fixture `05-withdrawal-decay-24h.json` requires distinguishing a withdrawal at h-24 from one at h-1; the engine cannot.

2. **No amount-weighted age (ageFactor).** Spec says "amount-weighted age" — engine uses `avgLiquidityDepthWeth` as a flat trailing average. Fixture `03-jit-lp.json` requires `ageFactor = 25/96 ≈ 0.26` for a JIT LP — the engine has no LP timeline at all.

3. **Slot-fairness fix missing.** Spec §6.9 says ageFactor denominator should be `max(1, token_age_hours)` (not fixed 96h). N/A right now because no ageFactor exists.

4. **Withdrawal-penalty α is in `ScoringConfig` (`recentWithdrawalPenalty`)** — spec calls it `LP_PENALTY_WINDOW × LP_PENALTY_TAU`. Naming alignment + version bump.

**Severity:** high. This is the most-divergent component. A real implementation needs an LP-event timeline (add/remove with timestamp + amount) on input, not an aggregate `recentLiquidityRemovedWeth`. Schema change + indexer wiring + scoring formula rewrite.

## 4. Retention (§6.4.4)

**Locked spec (per dispatch):**
- `retentionRatio × ageFactor`
- `RETENTION_DUST_THRESHOLD = 0.0001` (fraction of supply)

**Engine implementation:** `score.ts:374-392` (`computeRetention`).

**What matches:**
- Holder set intersection: `currentHolders ∩ retentionAnchor / retentionAnchor.size` (`score.ts:388-391`).
- Two-anchor weighted blend (long + short), default 60/40 (`types.ts:286-287`).

**What diverges:**

1. **No ageFactor.** Spec says `retentionRatio × ageFactor` — engine returns just the ratio. Slot-fairness denominator (§6.9) likewise missing.

2. **No dust threshold.** Holder sets are `Set<Address>` — no balance attached. A wallet that received a 1-wei airdrop counts as a "holder" if it shows up in both anchors. Spec says exclude balances < 0.0001 of supply. The `holderBalance` table in the indexer schema *does* carry balances; the projection layer would need to apply the threshold before forming the holder set.

3. **No EXCLUDED_HOLDERS application** (§41.3). The doc-comment at `types.ts:41` claims the holder sets are post-§41.3 filtering "upstream of scoring," but there is no actual filter wired today (see finding #15).

**Severity:** medium. ageFactor is the load-bearing change; dust threshold is mechanical.

## 5. Holder Concentration (§6.4 + §41) — ✅ matches

**Engine implementation:** `components.ts:31-46` (`computeHolderConcentration`).

**What matches:**
- HHI = 10000 × Σ(p_i²) where p_i = balance_i / Σ(balances) (`components.ts:36-43`).
- Mapping `score = 1 - log10(max(HHI,1)) / log10(10000)`.
- Reference table from spec §41.5 reproduced exactly:
  - HHI 10000 → 0.0
  - HHI 1000 → 0.25
  - HHI 100 → 0.50
  - HHI 10 → 0.75
  - HHI 1 → 1.0
- Track E `pipeline.py:hhi_score` is byte-equivalent.

**Caveat:** `holderBalances` is never populated by the indexer projection — see finding #15. Formula matches; input isn't wired.

**Severity:** none on the formula; deferred to projection layer.

## 6. Wallet attribution (§6.4.7) — ❌ divergent

**Locked spec (per dispatch):** `tx.from` is the attributed buyer; routers/aggregators are transparent.

**Engine + indexer reality:** `packages/indexer/src/V4PoolManager.ts:108` writes `taker: event.args.sender`. V4's `Swap` event emits `sender = address that called PoolManager.unlock()` — which is the universal router on a router-mediated trade, not the EOA. The doc comment at `V4PoolManager.ts:30-33` acknowledges this gap and tags it as "Track D — out of scope for this PR" (a prior PR). It is now in scope for Epic 1.22 per §6.4.7.

**What's needed:** swap rows must attribute to `event.transaction.from` (Ponder exposes this on `event.transaction.from`). For the rare AA case where `tx.from` is itself a smart wallet, tag the row `attribution = "fallback"` (per dispatch). Add an `attribution` column on `swap`.

**Severity:** high. This is the single biggest correctness fix on the indexer side for sybil-resistance — without it, all router-routed volume aggregates onto one address (the router), tanking effectiveBuyers and breaking velocity attribution.

## 7. EXCLUDED_TRADERS set (§6.4.7) — ❌ missing

`grep -rn "EXCLUDED_TRADERS\|EXCLUDED_HOLDERS" packages/` returns zero hits. The set should:
- Include all §41.3 EXCLUDED_HOLDERS addresses (V4 PoolManager, FilterLpLocker, FilterFactory, the 0xdEaD burn address, the SeasonVault, the POLVault).
- Include a maintained MEV-bot list (live).
- NOT include routers / aggregators (those are handled by `tx.from` attribution per finding #6).

**Where to put it:** new module `packages/scoring/src/excluded.ts` exporting `EXCLUDED_TRADERS: Set<Address>` with a `MEV_BOTS` env override (CSV). Indexer applies the filter in the projection layer (#15) before forming `volumeByWallet` / `currentHolders`.

**Severity:** medium-high. Without this filter, MEV bots and protocol addresses appear as legitimate buyers and inflate effectiveBuyers / velocity.

## 8. Component normalization (§6.7 fixed-reference) — ❌ divergent

**Locked spec:** each component's raw value is mapped to [0,1] via a versioned `*_REFERENCE` constant (e.g. `VELOCITY_REFERENCE = 90th-percentile of v5 cohort`). Same input → same score, regardless of cohort composition.

**Engine reality:**
- `packages/scoring/src/score.ts:70-72` — `normalizeMinMax` over the live cohort. Velocity / effectiveBuyers / stickyLiquidity each min-max'd to the cohort's own range.
- `packages/scoring/track-e/pipeline.py:182-185` — `rank(pct=True)` (percentile rank) over the corpus. Within-cohort percentile.

**Both are §6.7 violations.** A token with identical input bundle scores differently in a weak vs strong cohort — fixture `10-cohort-percentile-vs-score.json` explicitly forbids this.

**Why this matters:** under cohort-relative normalization, ranking is invariant to absolute quality (the bottom of a strong cohort can score the same as the top of a weak one). Under fixed-reference, HP is an *absolute* signal — a token's HP doesn't change just because peers improved.

**Refactor scope:**
- Add `VELOCITY_REFERENCE`, `EFFECTIVE_BUYERS_REFERENCE`, `STICKY_LIQUIDITY_REFERENCE` constants to `types.ts`. Calibrated values come from Phase 4 Track E v5 refit (90th percentile per §6.7).
- Replace `normalizeMinMax` with a pure mapping `raw → min(1, raw / REFERENCE)` per component.
- Mirror in `pipeline.py`.
- Preserve `cohortPercentile` as a separate display-only field on `/season` for the Arena tile view's mini-bars (per spec §19.6.1).

**Severity:** high. This is the single biggest behavioral change. **Must be gated by Track E v5 refit (Phase 4) — if the refit ρ falls outside `+0.36 ± 0.10`, fixed-reference is a regression and we re-think.**

## 9. Slot-fairness fix (§6.9) — ❌ missing

**Locked spec:** stickyLiquidity ageFactor + retention ageFactor should use `max(1, token_age_hours)` as denominator, not a fixed 96h. A slot-12 launch with a 24h-old maximally-sticky LP should be able to score 1.0 on stickyLiquidity within its window.

**Engine reality:** no ageFactor exists at all (see findings #3 + #4). Adding it correctly = adding the slot-fairness denominator from day one.

**Severity:** medium. Quasi-cosmetic until ageFactor exists; unblocks fairness for late-slot launches.

## 10. Cohort-edge cut semantics (§6.10) — ⚠️ partial

**Locked spec:** when fewer than 12 active tokens exist (deferred-activation aborted some), round down survivors and apply three-tier tie-break: HP desc → launchedAt asc → token-address asc.

**Engine reality:**
- HP desc ✅ — `score.ts:175-176`.
- launchedAt asc ✅ — `score.ts:177-181`.
- token-address asc ❌ — falls through to "stable Array.sort order" (`score.ts:182`) which is implementation-defined when DB-read order varies.

The round-down at <12 is not a scoring concern — it lives in the scheduler's cut decision (`packages/scheduler/src/...`) and the oracle's finalist-set publication. Out of scope for `scoring/` per se; document the cross-package contract.

**Refactor scope:** add a third sort tier (lowercase address ascending) to `score.ts:175-183`.

**Severity:** low. Probability of an exact-HP, identical-launchedAt collision is small but nonzero on a 12-token cohort with int10k resolution.

## 11. Multi-pool (§6.11) — ✅ effectively matches

**Locked spec:** `tokens.canonicalPoolId` should be the filter for all swap-side and LP-side queries.

**Engine reality:**
- The `pool` table is populated only from `FilterFactory:TokenDeployed` (`packages/indexer/src/FilterFactory.ts:12-19`) — exactly one pool per token.
- The V4 swap handler filters via the `pool` table (`V4PoolManager.ts:73-74`) — foreign pools / secondary pools never enter the indexed swap stream.
- The result is "canonical pool only" by construction, even though no explicit `canonicalPoolId` field exists on `tokens`.

**Recommendation:** promote the canonicality enforcement to an explicit field — `tokens.canonicalPoolId` (NOT NULL, FK → `pool.id`). Makes the contract self-documenting and lets future audit tooling assert "scoring read only canonical-pool data" mechanically. Low-risk refactor; the data is already there.

**Severity:** low. Already enforced; refactor is for clarity.

## 12. Reorg + finality (§6.12) — ❌ missing

**Locked spec:**
- `hpSnapshot.finality` field: `"tip" | "soft" | "final"`.
- Settlement-tagged rows (CUT, FINALIZE) wait ≥12 blocks past wall-clock boundary.
- Reorg rolls back affected rows + emits `HP_REORG` SSE event.

**Engine reality:**
- `hpSnapshot` schema (`ponder.schema.ts:211-255`) has no `finality` column.
- `recomputeAndStampHp` writes immediately on the trigger; no wait window.
- No `HP_REORG` SSE event in `events/types.ts`.

**Refactor scope:**
- Schema migration: add `finality TEXT NOT NULL DEFAULT 'tip'` to `hpSnapshot`.
- Writer logic: tag SWAP / HOLDER_SNAPSHOT / BLOCK_TICK rows as `tip`; transition to `soft` at +6 blocks, `final` at +12.
- CUT / FINALIZE writer: queue the row, write only after 12 confirmations past the wall-clock boundary.
- Reorg handler: on reorg detection (Ponder emits this), invalidate affected rows and emit `HP_REORG` SSE.

**Severity:** high for settlement correctness. Mainnet blocker — without the wait window, a reorg between CUT and the oracle's Merkle publish can ship a bad finalist set.

## 15. Indexer `tokenStatsFromRows` projection — ❌ stub (BIGGEST GAP)

**Engine reality:** `packages/indexer/src/api/hp.ts:43-57`:

```ts
export function tokenStatsFromRows(row: TokenRow): TokenStats {
  return {
    token: row.id as Address,
    volumeByWallet: new Map(),       // ← empty
    buys: [],                        // ← empty
    sells: [],                       // ← empty
    liquidityDepthWeth: 0n,          // ← zero
    currentHolders: new Set(),       // ← empty
    holdersAtRetentionAnchor: new Set(),  // ← empty
    launchedAt: row.createdAt,
  };
}
```

Doc comment at `hp.ts:5-12` self-documents this as a "known gap" deferred to a "future PR." No PR has landed yet. The result: every component (except `launchedAt` for tie-break) returns 0 / empty / zero in production. The engine is structurally correct but functionally inert.

**Why this matters for Epic 1.22:** the dispatch's acceptance criteria assume the engine is producing real HP scores. Once the indexer projection lands, formula divergences (#1-9) become *active behavior* — they affect real settlement. Right now the engine's bugs are dormant.

**Recommendation to Jake:** decide whether to:
- **(A) Ship Epic 1.22 as scoped** — formula lock + named constants + fixtures + finality. Document explicitly that the indexer projection is a separate gating epic before mainnet.
- **(B) Expand Epic 1.22** — fold in the projection (`swap` → `volumeByWallet` + `buys/sells`; `holderBalance` → `currentHolders` + `holdersAtRetentionAnchor`; LP timeline from V4 PoolManager `ModifyLiquidity` events → `liquidityDepthWeth` + per-event amount/timestamp stream for the new ageFactor). This is a substantial scope expansion (~3-5 days additional engineering on top of the current dispatch).
- **(C) Stage in two PRs** — PR 1: formula lock + fixtures + named constants + finality (this dispatch). PR 2: indexer projection (sub-epic 1.22b).

Recommend (C). The formula lock + fixtures are independently auditable and the version bump is meaningful even with empty inputs (it tags future rows correctly). The projection PR is then mechanically validatable against the new fixtures.

**Severity:** critical for production end-to-end behavior. Not blocking for the formula-lock PR if we choose (A) or (C).
